ISSUE #1: RESOLUTION CHANGE SYSTEM
-----------------------------------
SYMPTOM: User can change resolution in settings, but screen doesn't visually change
CLICKS REGISTERED: YES (Option button responds)
VISUAL RESULT: NO (Window doesn't resize visually)

ROOT CAUSES IDENTIFIED:
1. Fullscreen Mode Lock (PRIMARY)
   - Game starts in FULLSCREEN mode (project.godot: size/mode="fullscreen")
   - When in fullscreen, OS prevents window.size changes
   - The resize command executes but has no visual effect

2. Insufficient Timing After Fullscreen Exit
   - Code exits fullscreen with only 1 frame wait
   - Fullscreen transition may need more time to complete
   - Resize happens too quickly after exit

3. Stretch Mode Interference
   - Stretch mode "canvas_items" may affect viewport scaling
   - Different OS/driver combinations handle this differently

VERIFICATION:
  ✓ _on_resolution_selected() method exists with correct logic
  ✓ Signal connection properly set up
  ✓ OptionButton has 5 resolution options populated correctly
  ✗ ISSUE CONFIRMED: Fullscreen prevents resize visibility


ISSUE #2: OVERHEAT UI SYSTEM
-----------------------------
SYMPTOM: Clicks register (CPU increases) but overheat bar doesn't update visually
CLICKS REGISTERED: YES (Console shows CPU increasing)
VISUAL RESULT: NO (Bar and text don't change)

ROOT CAUSES IDENTIFIED:
1. @onready Variable Null Issue (PRIMARY)
   - Location: res://ui/cpu_hud.gd lines 18-20
   - CPUHUD scene is instantiated at runtime in main.gd line 29
   - @onready variables resolve at script attachment time
   - Before scene is added to tree, node paths don't resolve
   - Variables silently become null
   - Update method receives signal but can't modify null references

2. No Initialization Fallback
   - No manual initialization in _ready()
   - No error handling for null references
   - Silent failures when trying to update null objects

3. Missing Null Checks
   - Update methods assume UI elements exist
   - No validation before using them
   - Runtime errors handled silently

4. Possible Theme Property Incompatibility
   - ProgressBar fill color override might use wrong property name
   - "fill" property name may not match current Godot version

VERIFICATION:
  ✓ Method _on_overheat_updated() exists with correct logic
  ✓ Signal connection properly registered
  ✓ Player emits signal correctly
  ✓ Scene structure has all UI elements
  ✗ ISSUE CONFIRMED: @onready variables null due to runtime instantiation
  ✗ No fallback initialization in _ready()
  ✗ No null checks before using UI references

TECHNICAL DETAILS
================================================================================

RUNTIME INSTANTIATION ISSUE EXPLANATION:
- main.gd line 28-30: cpu_hud_scene.instantiate() is called at runtime
- Before add_child(), the scene is not in the tree
- @onready variables resolve based on relative paths from the script location
- If the script is not in the tree yet, relative paths fail
- Solution: Manual initialization after add_child() in _ready()

FULLSCREEN TIMING ISSUE EXPLANATION:
- get_window().mode = Window.MODE_FULLSCREEN is asynchronous on many OS
- The window manager handles the transition
- Setting window.size immediately after exiting fullscreen may fail
- The resize request is queued but executed before the transition completes
- Solution: Use await get_tree().process_frame multiple times (3+ frames)

THEME PROPERTY COMPATIBILITY:
- Different Godot versions use different property names
- ProgressBar.fill might be called "fill", "fill_color", or modified via modulate
- Solution: Try "fill" first, fallback to modulate if not available
